# Incident Report: Cross-Site Scripting (XSS) Attack Attempt — July 22, 2026

## Executive Summary

On July 22, 2026, at approximately 07:55 UTC, our security systems detected and blocked a series of Cross-Site Scripting (XSS) attacks targeting our web application. An attacker, operating from within our internal network, attempted to inject malicious JavaScript code into our product search, feedback, and user management pages. Our Web Application Firewall (ModSecurity with the OWASP Core Rule Set) successfully intercepted all six attack attempts, returning a 403 Forbidden response each time. No data was compromised, no customer information was exposed, and no unauthorized code execution occurred. This report details what happened, how our defenses responded, and the steps we are taking to further harden our application against future threats.

## Background

Cross-Site Scripting, commonly known as XSS, is a type of security vulnerability that allows an attacker to inject malicious scripts into web pages viewed by other users. In a typical XSS attack, the attacker finds an input field — like a search box or a feedback form — that does not properly validate or encode user input. They then submit a string containing JavaScript code. If the application does not sanitize this input before displaying it on a page, the attacker's script can run in the browser of anyone who visits that page. This can be used to steal session cookies, redirect users to phishing sites, or perform actions on behalf of the victim.

The affected systems in this incident were our web application's search endpoint (`/rest/products/search`), the feedback submission API (`/api/Feedbacks`), and the user management API (`/api/Users`). These are standard components that accept user input to perform their functions. Our security architecture includes a Web Application Firewall (WAF) running ModSecurity with the OWASP Core Rule Set, which inspects all incoming HTTP requests before they reach the application server. The WAF uses a technique called libinjection to detect SQL injection and XSS attempts by analyzing the syntactic structure of input data. All traffic is also monitored by our Wazuh SIEM platform, which collects and correlates security events in real time.

## Incident Timeline

**2026 July 22 07:55:21 UTC**

The attacker launched their first probe, sending a GET request to our product search endpoint. The query parameter `q` contained a classic XSS payload: a script tag designed to execute JavaScript that would read and exfiltrate the user's session cookie. Our WAF immediately recognized the injection pattern using libinjection analysis and returned a 403 Forbidden response. The request never reached the application server. The WAF rule triggered was 941100, specifically designed to detect XSS attacks through contextual analysis of the input data.

**2026 July 22 07:55:40 UTC**

Undeterred by the first block, the attacker sent a second request to the same search endpoint, this time using an HTML image tag with an `onerror` event handler. This is a common evasion technique that attempts to bypass simple keyword filters by hiding the JavaScript within an event attribute. The payload `<img src=x onerror=alert(1)>` would have executed JavaScript if the image failed to load — which it always would, since `src=x` is not a valid image URL. Our WAF's libinjection engine again detected the attack, flagging the input as containing executable script content and blocking the request with a 403 status.

**2026 July 22 07:55:50 UTC**

The attacker tried yet another variant, this time using an SVG-based vector. The payload `<svg/onload=alert(document.domain)>` attempted to execute JavaScript when the SVG element finished loading. This technique is sometimes used to bypass filters that look for specific HTML tags like `script` or `img`. Once again, ModSecurity's libinjection analysis detected the injection pattern — the combination of a markup context with executable event handlers triggered the XSS rule, and the request was blocked. The attacker had now attempted three different payload types against the same endpoint in under thirty seconds, all unsuccessful.

**2026 July 22 07:56:17 UTC**

Shifting tactics, the attacker changed from GET requests to POST requests and moved to a different endpoint: the feedback submission API. POST requests are used to submit data to the server, and in this case, the attacker was attempting a stored XSS attack. Unlike reflected XSS (which executes immediately in the response), stored XSS places the malicious code into the application's database so that it executes later when other users view the feedback page. This type of attack is often considered more dangerous because it can affect many users without requiring them to click a specially crafted link. The WAF inspected the POST body content, detected the XSS payload, and returned a 403 Forbidden. The attacker's attempt to persist malicious code in our system was stopped at the perimeter.

**2026 July 22 07:56:35 UTC**

The attacker sent a second POST request to the feedback endpoint, presumably with a different payload variation in an attempt to evade detection. The WAF again identified the XSS attempt through libinjection's structural analysis of the input, which looks at the grammar and syntax of the submitted data rather than relying on static blacklists. The request was blocked. This was the second attempt to plant a persistent XSS payload through the feedback system, and the second time our defenses held firm.

**2026 July 22 07:56:52 UTC**

In a final attempt, the attacker escalated their efforts to the user management API (`/api/Users`). This endpoint likely handles user registration or profile updates and represents a high-value target — successful exploitation could allow the attacker to inject script code that would execute in the context of administrative user sessions. The WAF intercepted this POST request as well, flagging it with the same 941100 rule and returning a 403 response. With this last block, the attacker's activity ceased. The entire attack sequence lasted just over one and a half minutes, encompassing six distinct attack attempts across three different endpoints, all of which were successfully neutralized by our security controls.

## Root Cause Analysis

This incident was a targeted XSS attack that exploited the way our web application handles user-supplied input. The root cause lies in the application's design: user input fields — specifically the product search query parameter, the feedback submission form, and the user management API — accepted raw input from users without performing adequate validation, sanitization, or encoding at the application layer. In a properly secured application, user input should be treated as untrusted data and either escaped before being rendered in HTML or validated against a strict allowlist of safe characters.

The specific trigger in this case was the attacker's XSS payloads, including script tags (`<script>alert(document.cookie)</script>`), HTML event handlers (`<img src=x onerror=alert(1)>`), and SVG-based vectors (`<svg/onload=alert(document.domain)>`). These attacks targeted the trust relationship between the application and its users' browsers — the application would normally reflect search queries or display submitted feedback content back to users, and the attacker attempted to inject executable code into that content.

The vulnerabilities that allowed this attack to be attempted — though ultimately not exploited — were twofold. First, the application lacked input sanitization at the code level. The endpoints were accepting raw input without stripping or encoding HTML and JavaScript content. This is a common development oversight, often resulting from prioritization of feature delivery over security hardening. Second, while our WAF provided an effective protective layer, the application should not rely solely on a perimeter defense for input validation. Defense in depth requires that the application itself also perform input sanitization as a fallback should the WAF ever be misconfigured, bypassed, or encountered with a novel attack vector.

Why did this happen? Our development lifecycle had not yet integrated automated security testing for XSS vulnerabilities, and the input sanitization controls that were planned for a future sprint had not been implemented. The attacker likely performed reconnaissance on our application — possibly testing input fields with benign characters first — then escalated to XSS payloads once they identified the lack of client-side validation. Fortunately, our WAF infrastructure acted as a safety net, but the incident highlights a gap in our application security posture that could have been more severe if the attacker had used a more sophisticated payload or targeted a WAF blind spot.

## Learnings

This incident has reinforced the importance of a layered security approach. While our WAF performed exactly as designed — detecting and blocking all XSS attempts with no false negatives — we cannot become complacent. Our immediate actions include implementing input sanitization and output encoding at the application layer for all user-facing endpoints, particularly the search, feedback, and user management functions that were targeted. We are also accelerating our integration of automated security scanning into the CI/CD pipeline, ensuring that XSS vulnerabilities are detected during development rather than in production. Finally, we are conducting a thorough review of our internal network access controls, given that the attack originated from an internal IP address. We are committed to transparency about these events and to continuously improving our security posture to protect the trust our users place in us.
