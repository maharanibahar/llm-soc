## Executive Summary

On July 22, 2026, our Web Application Firewall detected and blocked a series of reconnaissance attempts from an internal IP address. The attacker tried to retrieve sensitive files such as `.env` and `.git` directories from one of our web servers. These files can contain confidential credentials, API keys, or source code if improperly exposed. Our defense systems responded immediately, stopping all three requests before any information could be accessed. No data breach occurred, and no systems were compromised. This incident highlights the effectiveness of our security controls and reaffirms our commitment to layered protection.

## Background

The attack we observed is a common technique known as **file-disclosure reconnaissance**. Attackers probe web servers for known sensitive files that may have been inadvertently left accessible. For example, a `.env` file is used by many applications to store environment variables such as database passwords and secret keys. Similarly, the `.git` directory contains the full version history of a project, including source code and sometimes embedded credentials. If such files are served by the web server without restriction, they can leak valuable information.

The targeted system is an internal web application protected by a Web Application Firewall (WAF) running ModSecurity, which is integrated with our Wazuh security monitoring platform. The WAF inspects every HTTP request and applies a set of rules to block known attack patterns. In this case, the rule designed to prevent access to restricted system files (Rule 930130) immediately flagged and blocked the attempts. Our security posture is further strengthened by centralized logging and real-time alerts, which enable our team to review and respond to threats quickly.

## Incident Timeline

### 2026 July 22 07:43:59 UTC

An automated scanner, originating from IP address 172.19.0.12 within our internal network, sent an HTTP GET request for the path `/.env`. This file is a standard location for environment configuration and is a prime target for credential theft. The request reached our WAF before touching the application server. ModSecurity evaluated the request against its rule set and identified it as a “Restricted File Access Attempt.” The WAF immediately blocked the request with a 403 Forbidden response and logged the event. No content was returned to the requester.

### 2026 July 22 07:44:00 UTC

One second later, the same IP requested `/.git/config`. The `.git` directory holds deep metadata about source code, and the `config` file can reveal repository URLs and authentication tokens. Once again, our WAF intercepted the request, applied rule 930130, and denied access. The attacker received no useful information and the incident was recorded.

### 2026 July 22 07:44:01 UTC

The final attempt targeted `/.git/HEAD`, another file that helps attackers confirm the existence of a Git repository and identify the active branch. The WAF responded identically, blocking the request and logging the event. The three attempts occurred within a three-second window, confirming the organized, automated nature of the probe. After the third block, no further attempts were observed from this source IP.

## Root Cause Analysis

The trigger for this incident was deliberate scanning for exposed sensitive files. The attacker specifically chose paths that are common weak points in web application security: `.env` for credentials and `.git` for source-code disclosure. While these paths are valid only when misconfigured (i.e., when a web server serves static files outside its intended scope), our WAF blocked them regardless of whether the files actually existed. The gap that allowed the probe to reach our perimeter was not a vulnerability in the application itself, but rather the inherent risk of exposing any web resource to internal network traffic. The attacker exploited a **lack of network segmentation** or **insufficient internal access controls** to reach the server, but our defense-in-depth compensated by halting the attack at the application layer.

Importantly, the attack did not succeed. Our WAF rule 930130 (Restricted File Access Attempt) acted as the intended checkpoint. The root cause of the incident being notable is that it demonstrates how a determined internal actor or compromised host could attempt to gather intelligence. Our security layers performed exactly as designed, preventing any impact.

## Learnings

This incident reinforces the value of a properly tuned Web Application Firewall and the importance of continuous monitoring. The automated detection and blocking prevented what could have been a serious information disclosure. Moving forward, we will review our internal network segmentation to reduce the attack surface further and consider adding additional WAF rules to cover similar file paths. We have also scheduled a broad scan of all internal web servers to ensure no `.env` or `.git` artifacts exist in public directories. We remain committed to transparency with our users and stakeholders, and we will continue to invest in security controls that protect against both external and internal threats.
