## Executive Summary

On 22 July 2026, our security team detected a series of attempted SQL injection attacks against our public-facing web application. The attacker, originating from a single internal IP address (172.19.0.10), targeted our product search and user login endpoints with a variety of common SQL injection payloads. Thanks to our Web Application Firewall (WAF), every single attempt was automatically identified and blocked within seconds. No sensitive data was accessed, no user accounts were compromised, and the application remained fully available throughout the incident. This report details what happened, why our defenses were effective, and what we have learned to further strengthen our security posture.

## Background

SQL injection is a type of cyberattack where a malicious actor inserts or “injects” malicious SQL code into a website’s input fields—like a search box or login form. If a website does not properly validate or sanitize user input, this injected code can trick the database into revealing confidential information, such as user passwords or credit card numbers, or even allow the attacker to take control of the database. It is one of the oldest and most common threats to web applications.

The systems affected in this incident are our customer-facing product search and login modules. Both rely on a backend database to retrieve product listings and verify user credentials. To protect these systems, we deploy a Web Application Firewall (WAF) that sits between the internet and our application servers. The WAF inspects every incoming request for known attack patterns—including SQL injection—and blocks anything suspicious before it ever reaches the application code. In this case, the WAF was our first and strongest line of defense.

## Incident Timeline

### 2026 July 22 07:39:01 UTC
The attacker began by sending a GET request to our product search endpoint (`/rest/products/search`) with the parameter `q=lol'+OR+1=1--`. This classic SQL injection attempts to make the database query always return true, which could allow the attacker to see all products or bypass access controls. Our WAF immediately recognized this as a SQL injection attack (rule 942100) and returned a 403 Forbidden response, blocking the request.

### 2026 July 22 07:39:02 UTC
Less than one second later, the attacker escalated the attempt by sending a more advanced payload: `lol'+UNION+SELECT+1,2,3,email,password,6,7,8,9+FROM+Users--`. This UNION-based injection aims to extract the `email` and `password` columns from a table called `Users`. The WAF again detected the malicious intent and blocked it with a 403 response.

### 2026 July 22 07:39:04 UTC
The attacker changed tactics, using a boolean-based blind SQL injection: `lol'+AND+1=CONVERT(int,(SELECT+TOP+1+email+FROM+Users))--`. This technique tries to infer information one bit at a time by causing a conversion error if a condition is true. The WAF’s libinjection engine identified the pattern and blocked the request.

### 2026 July 22 07:39:05 UTC
A time-based blind injection followed: `lol'+(SELECT+SLEEP(5))--`. This payload would force the database to pause for 5 seconds if executed, allowing the attacker to confirm a vulnerability even when no error messages are shown. Once more, the WAF blocked it immediately.

### 2026 July 22 07:39:06 UTC – 07:39:07 UTC
Finally, the attacker shifted focus to the user login endpoint (`/rest/user/login`), sending two POST requests. While the exact payloads are not detailed in the logs, the WAF flagged both as SQL injection attacks and returned 403 responses. These last attempts also failed to reach the application.

The entire attack lasted only seven seconds. The WAF blocked all six requests without any manual intervention, and no further activity from this IP was recorded.

## Root Cause Analysis

The root cause of this incident is simple: a malicious actor deliberately attempted to exploit potential SQL injection vulnerabilities in our web application. The **target** was the product search and user login functions, which accept user-supplied input that could be used to construct database queries. The **trigger** was a series of well-known SQL injection payloads, including tautologies (`OR 1=1`), UNION-based data extraction, and blind injection techniques (boolean and time-based).

Our investigation did not find any evidence that the application code itself had a SQL injection vulnerability—the WAF intercepted the attacks before they could be tested against the database. However, the attacker’s choice of payloads suggests they were probing for a common weakness: unfiltered user input passed directly into SQL statements. The **gap** that allowed this incident to happen is the inherent risk of relying solely on a perimeter defense (the WAF) without also enforcing input validation and parameterized queries at the application level. While the WAF performed perfectly, we acknowledge that defense-in-depth—multiple layers of security—is always better than a single control.

## Learnings

This incident reinforces the value of our Web Application Firewall and the importance of continuous monitoring. Our security team has reviewed the event and confirmed that no data was compromised and no service was disrupted. To further harden our defenses, we are conducting a thorough code review of the product search and login modules to ensure that input validation and prepared statements are used consistently. We are also adding additional WAF rules to detect and block similar patterns even more rapidly. We believe in transparency with our users: the attack was detected and neutralized, but we are using this opportunity to make our systems even more resilient. We remain committed to protecting your data and maintaining your trust.
