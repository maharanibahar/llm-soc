# Incident Report: Sensitive File Enumeration Attempt (Successfully Blocked)

**Report Date:** 2026-07-23  
**Report Author:** Security Operations Team

## Executive Summary

On July 22, 2026, between 07:59 and 08:00 UTC, an automated scanner attempted to access common sensitive files on our web application. The attacker probed for environment variable files (`/.env`) and Git repository configuration files (`/.git/config` and `/.git/HEAD`), likely seeking credentials, API keys, or source code metadata. All three attempts were immediately detected and blocked by our Web Application Firewall (ModSecurity). No sensitive data was exposed, and no systems were compromised. The incident demonstrates the effectiveness of our existing security controls and the importance of layered defenses.

## Background

This type of attack is called **information disclosure reconnaissance**. Attackers scan thousands of websites for unprotected files that are often left accessible by mistake. For example, `.env` files commonly store database passwords, API tokens, and other secrets used by the application. The `.git` folder contains the entire version control history, including source code and previous secrets. When these files are publicly accessible, an attacker can extract them and use the information to launch further attacks. Many automated tools exist to find such files, and these scans happen continuously across the internet.

Our web application sits behind a reverse proxy that runs ModSecurity, a widely-used Web Application Firewall (WAF). We have configured a set of rules to block known attack patterns, including the "Restricted File Access Attempt" rule (ID 930130). This rule specifically monitors for requests that target files like `.env`, `.git`, `.htaccess`, and other sensitive paths. Our detection pipeline also forwards WAF logs to the Wazuh SIEM platform, providing real-time visibility to our security team.

## Incident Timeline

### 2026 July 22 07:59:21 UTC
The attacker's first request targeted the `/.env` path. ModSecurity immediately evaluated the URL against its rule set. Rule 930130 matched because the string `/.env` is recognized as a restricted file pattern. The WAF returned a 403 Forbidden status to the attacker and logged the event with the message "Restricted File Access Attempt." The connection was terminated before reaching the web server.

### 2026 July 22 07:59:36 UTC
Fifteen seconds later, the attacker sent a second request to `/.git/config`. The WAF again triggered rule 930130. This time, the request also included a pattern indicating the attacker was specifically interested in Git repository configurations. The WAF blocked this attempt as well, returning a 403 response and recording the event.

### 2026 July 22 07:59:55 UTC
The third and final request targeted `/.git/HEAD`, another critical file inside the Git metadata directory. Once more, ModSecurity applied the same rule and blocked the request. The WAF action `block` ensured the attacker received no data. The entire probe lasted less than a minute and failed to retrieve any sensitive information.

## Root Cause Analysis

The attack itself was a straightforward automated scan – a common occurrence on public-facing web applications. The **target** was our web application's configuration files. The **trigger** was a set of HTTP GET requests for well-known sensitive paths. The attacker exploited the *potential* vulnerability of these files being accidentally left exposed.

From the attacker's perspective, the success of such a scan relies on absent or ineffective access controls. In our environment, the **root cause gap** that would allow a successful exploit would be a web server configured to serve static files without restrictions, or the absence of a WAF rule to block these patterns. However, in this incident, our defences worked exactly as designed. We have rule 930130 enabled on our ModSecurity instance, which specifically prevents any attempt to access restricted files. This rule was the direct countermeasure that stopped the attack.

No software vulnerability was involved – this is purely an attack against a common configuration weakness. The exposure risk is mitigated by both the WAF and by the eventual file permissions and server configuration that we maintain. In this case, the WAF was the first line of defence and successfully prevented any further interaction with the application.

## Learnings

This incident confirms that our security monitoring and WAF rules are functioning as expected. The automated scan was detected and blocked without requiring human intervention. However, we recognize that no defense is perfect, and we must remain vigilant. We will conduct a review of our web server configuration to ensure that sensitive files are not only blocked by the WAF but also inaccessible at the filesystem level. We will also enhance our WAF logging to capture additional context, such as User-Agent headers, when present. Additionally, we will use this event to fine-tune our SIEM alerting for low-and-slow reconnaissance patterns. Our commitment to transparency means we share this report with our users to demonstrate both our proactive monitoring and our willingness to openly discuss security events, even those that did not result in harm. We will continue to invest in layered defences so that similar – or more sophisticated – attempts are caught just as effectively.
