## Executive Summary

On July 22, 2026, our security systems detected and blocked a series of attempted Cross‑Site Scripting (XSS) attacks against the company’s web application. Between 07:39:08 and 07:39:14 UTC, an attacker launched six probing requests aimed at injecting malicious scripts into user‑input fields. All attempts were stopped immediately by our Web Application Firewall (WAF), and no data was compromised, no unauthorized access occurred, and no service degradation was experienced. The incident was fully contained within seconds.

## Background

Cross‑Site Scripting (XSS) is a type of attack where a perpetrator injects malicious scripts into otherwise trusted websites. When a user visits an affected page, the script can run in their browser, potentially stealing cookies, session tokens, or redirecting them to harmful sites. The attack is most commonly carried out by placing JavaScript code in form inputs or URL parameters that the web application later echoes back without proper sanitization.

The attempted breach targeted three areas of our application: the product search endpoint, the feedback submission form, and the user registration page. These are all features that accept content from end‑users. Thankfully, our security architecture includes a Web Application Firewall (ModSecurity) placed in front of the application server. This WAF inspects every incoming request and compares it against known malicious patterns, including the XSS patterns used in this incident.

## Incident Timeline

### 2026 July 22 07:39:08 UTC
The attacker started with a GET request to `/rest/products/search` with the query parameter `q` containing `<script>alert(document.cookie)</script>`. This is a classic XSS payload that, if stored or reflected, would have attempted to read and expose cookies from other users’ sessions. The WAF recognized the injection attempt and immediately returned a 403 Forbidden status, blocking the request.

### 2026 July 22 07:39:09 UTC
A second GET request was sent to the same search endpoint, but this time with an `<img>` tag using an `onerror` handler: `<img src=x onerror=alert(1)>`. This technique tries to execute JavaScript even when the `<script>` tag is blocked. Again, the WAF detected the XSS pattern (rule 941100) and responded with a block.

### 2026 July 22 07:39:11 UTC
The attacker switched to another vector, sending `<svg/onload=alert(document.domain)>`. SVG elements can sometimes bypass rudimentary filters. Our WAF, however, was not fooled; it matched the payload as XSS and repeated the 403 response, keeping the application safe.

### 2026 July 22 07:39:12 UTC
After the three search‑based probes, the attacker changed tactics and sent a POST request to `/api/Feedbacks`. The request body likely contained script code, attempting to place malicious content into the feedback database. The WAF examined the POST body and again blocked the request before it could reach the application.

### 2026 July 22 07:39:13 UTC
A third attempt against the feedback endpoint followed, using a slightly different XSS payload. Our defense continued to function perfectly, blocking the attempt with the same rule.

### 2026 July 22 07:39:14 UTC
The final logged attempt was a POST request to `/api/Users`, the registration endpoint. The attacker apparently hoped to inject JavaScript into a user profile field. Once more, the WAF intercepted the request and denied it. After this sixth attempt, the activity ceased.

## Root Cause Analysis

The attack was an attempt to exploit potential XSS vulnerabilities in our web application’s input‑handling code. The **target** was any endpoint that accepts user‑supplied data, specifically `/rest/products/search`, `/api/Feedbacks`, and `/api/Users`. The **trigger** was the set of crafted HTTP requests containing JavaScript payloads in the `q` parameter or POST body.

Several factors allowed the attacker to try these injections:
- **Lack of strict input validation at the application layer** – The endpoints did not appear to reject or sanitize the metacharacters necessary for XSS (e.g., `<`, `>`, `(`, `)`). Though the WAF blocked them, the application itself would not have safely handled these inputs.
- **Absence of rate limiting** – The attacker was able to send six requests within seven seconds without being throttled, indicating that rapid‑fire probing was not impeded.
- **Internal source IP** – The requests originated from a private IP address (172.19.0.11), which raises the possibility of an insider threat or a compromised internal machine. This vector bypasses external perimeter defenses and must be investigated further.

The root cause, then, is a combination of insufficiently hardened application input handling and a lack of internal traffic controls. The WAF served its purpose, but the incident highlights the need to strengthen the security layers behind it.

## Learnings

This incident reaffirmed the value of our defense‑in‑depth strategy, particularly the WAF that prevented a near‑certain compromise of user data. At the same time, it revealed clear areas for improvement: our application endpoints should not rely solely on a perimeter device to sanitize input, and we must implement internal access controls to monitor and limit unexpected internal traffic. We have already begun adding server‑side input validation and output encoding to the affected endpoints, and we are deploying additional monitoring on internal network traffic. We are committed to transparency about our security posture and will continue to share what we learn as we harden our systems against future attacks.
